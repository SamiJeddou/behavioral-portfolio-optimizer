- Added `mv_frontier_weights(means, cov, n_points)` — weights along the long-only Markowitz efficient frontier (GMV return → max single-asset return), for plotting the classical bullet.

### `app.py` — Grid Optimiser unified interactive 3D landscape
- `_grid_obj_surface_view()` (a `@st.fragment`) is a **unified, configurable** 3D landscape with **three selectboxes**: Instrument — X axis and Instrument — Y axis (each chosen from **securities + the derivative**), and Objective (**Mean-variance** E[r]−½λ·var / **Downside-utility** E[r]−κ·CVaR / **Expected return**). Renders a rotatable Plotly 3D Surface over the two chosen instruments' weights (others held at the optimum's proportions, remainder fills the long-only simplex), **RdYlGn colorscale** (green = high/optimum, red = low/bad — so the peak reads as "good", matching convention; also aligns with positive/negative objective values) + "Objective" colorbar, dark theme. Optimum marker has a dark outline so it stays visible on the green peak. Shown expanded-by-default in the results; title above the Summary frame.
- **Downside-utility is gated to require the derivative on an axis.** When both axes are securities (or no derivative is configured) the objective list is only Mean-variance + Expected return; the moment the derivative is on the X or Y axis, Downside-utility appears (and the objective state falls back to Mean-variance if the prior choice is no longer offered). Rationale: on a securities-only slice the derivative is frozen in the background, so the optimum wouldn't be the peak — hiding the option there avoids a misleading off-peak marker. Method note aligned to state this.
- **The optimum marker uses the EXACT grid weights** from the results: **Portfolio (2)** (amber) when the derivative carries weight, else **Portfolio (1)** (green) — so the marker's coordinates always match the summary. **It is a *constrained* optimum, so it generally sits just *inside* the feasible boundary, a touch below the surface's own peak** (the unconstrained peak is a more aggressive portfolio that breaches the downside limit, so the engine can't pick it — that gap is by design). The one case that lands exactly on a peak is Mean-variance with Portfolio (1) (no derivative, a genuine unconstrained hill-top). The caption and the "Method & approximations" note state this plainly so the off-summit placement reads as correct, not a bug. (Decision: keep honest exact-coordinate placement rather than snapping the marker to the surface peak.)
- **Approach & approximations (documented in-app via an HTML `<details>` "Method & approximations" note, and here):** the surface is a **Monte-Carlo illustration**, not the exact grid state-space solve — securities are drawn from a Gaussian copula on the estimated means/covariances and the derivative is Black-Scholes-priced on its underlying (the *scalable* engine's model), so the surface can differ slightly from the exact grid. κ for the downside-utility is a heuristic calibration (κ = ½·λ·var ÷ |CVaR| at the optimum), so that peak is approximate, not the engine's exact shadow price.
- During grid compute, stores `st.session_state['_grid3d_data']`: instrument **labels**, a joint **Monte-Carlo scenario matrix `R` (S×M, float32)** built with `mc_generate_scenarios` + `mc_build_matrix` (derivative spec from the grid's `der_type`/`der_params`; falls back to securities-only if pricing is incompatible), `n_sec`, Portfolio (1) weights `w1`, Portfolio (2) weights `w2` (extra `run_opt` with the derivative), implied λ, and the constraint (H, α/L). Works on both fresh-compute and cache render paths.
